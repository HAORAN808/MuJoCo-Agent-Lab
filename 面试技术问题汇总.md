# Demo 面试技术问题全量准备手册

适用方向：智能实验室与实验操作系统、智能体赋能科学发现、AI for Science 工具链、科研自动化平台、实验闭环智能体。

项目当前定位：**面向机器人操作实验的 AI 原生实验闭环原型**。它把自然语言研究目标转化为结构化实验计划，调用 MuJoCo 仿真环境作为可编程实验仪器，批量执行实验，回收结构化日志、轨迹和物理指标，再由智能体完成失败归因和下一轮实验建议。

面试叙事核心：本项目是一个以机器人操作实验为载体的实验工作流 Agent 原型，重点展示“实验可编程、执行可验证、结果可回流、失败可归因、策略可迭代”的科研实验自动化链路。

---

## 0. 面试核心叙事

本项目的讲解重点不是“机械臂能不能动”，而是“一个实验任务如何被 Agent 组织成可执行、可验证、可迭代的闭环”。面试官会同时关注三层能力：

- 是否理解科学实验自动化的核心抽象；
- 是否能把 Agent、工具、协议、执行器、评估和记忆组织成工程系统；
- 是否能诚实说明当前 Demo 的边界，并给出可落地的演进路线。

项目可以用下面这张结构表来介绍：

| 实验智能体能力 | 本项目实现 |
| --- | --- |
| 目标理解 | 自然语言实验目标解析 |
| 协议生成 | TaskSpec、ExperimentPlan、SkillPlan |
| 工具调用 | MuJoCo 虚拟实验仪器 |
| 物理执行 | 动作原语、IK、刚体接触仿真 |
| 结果评估 | EvaluationReport、目标误差、物理安全 |
| 失败归因 | no_contact、undershoot、overshoot、table_penetration 等 |
| 自动迭代 | RetryPlan、AutoRetry |
| 经验沉淀 | ExperimentStore |

推荐的项目定位：

> 当前 Demo 不是一个端到端机器人控制 Agent，而是一个实验工作流 Agent。它把自然语言实验目标转成结构化协议，调用 MuJoCo 执行物理实验，回收指标和失败类型，再自动执行一轮修正实验并写入实验记忆。它验证的是科研实验自动化中的“协议化、工具调用、数据回流、失败归因和迭代决策”链路。

表达边界：

- 当前系统是受约束的 agentic workflow，不是裸 LLM 控制机器人。
- LLM 不直接控制硬件，底层通过 SkillPlan、Adapter 和安全验收执行。
- 当前证明的是闭环可行性，泛化能力需要更多任务、统计验证和真实硬件接入。
- 成功率必须绑定样本数、变量空间和成功标准解释，不能被表述成策略泛化能力。

## 1. 最高频开场问题

### Q1：请用一分钟介绍你的 Demo 项目。

我做的是一个面向机器人操作实验的智能实验闭环系统。用户输入自然语言研究目标，比如“研究不同摩擦和位姿扰动下机械臂抓取成功率”，系统会把目标解析成结构化任务规格，自动选择机器人、物体、动作序列和实验变量，调用 MuJoCo 作为可编程实验环境批量执行实验，然后回收成功率、失败类型、物体位移、接触步数、最大接触力和轨迹回放，最后生成失败归因和下一轮实验建议。

它的重点不是训练一个最优机器人策略，而是验证科研实验能不能被智能体组织成“目标 - 假设 - 实验设计 - 执行 - 数据回流 - 分析 - 下一轮决策”的闭环。

### Q2：这个项目为什么和智能实验室方向相关？

智能实验室的核心是打通实验流程、仪器控制和数据回流。我的项目里，MuJoCo 扮演可编程实验仪器，Python API 扮演仪器控制层，结构化 logs 和 metrics 扮演数据回流层，agent 负责目标理解、实验设计和下一轮决策。

所以它虽然当前载体是机器人 MuJoCo 实验，但抽象上对应的是智能实验室操作系统：把实验变成可编程、可验证、可迭代的执行与反馈层。

### Q3：这个项目为什么和科研智能体方向相关？

科研智能体要做的是把知识、模型、工具和实验接口组织成可执行工作流，并根据结果持续推进研究。我的系统中，用户给出研究目标后，智能体会解析目标、调用 MuJoCo 工具、执行实验、读取结果、分析失败并提出下一轮实验。这和科研智能体从目标到工具调用再到实验反馈的链路是一致的。

如果放到更大的科研平台里，我这个系统可以作为一个 domain tool，也就是“机器人操作实验执行器”。

### Q4：你希望面试官记住这个项目的哪三个关键词？

第一是 **实验闭环**：不是单次仿真，而是从研究目标到下一轮实验建议。

第二是 **可验证执行**：LLM 不直接控制机器人，而是在 MuJoCo、runner、动作原语和指标引擎约束下执行。

第三是 **智能实验室抽象**：MuJoCo 是虚拟实验仪器，runner 是实验协议，logs/metrics 是数据回流，agent 是科研工作流编排器。

## 2. 项目价值与问题背景

### Q5：你为什么做这个项目？

机器人算法研发里，很多时间不是花在单次运行，而是花在调参、复盘和定位失败边界上。传统方式依赖人工记录和经验判断，常见问题是实验记录不结构化、失败类型难复用、下一轮实验设计靠直觉。

我想验证一个想法：能不能让智能体承担“研发实验员”的角色，把研究目标拆成假设和变量空间，自动调用实验环境执行，再基于数据提出下一轮实验建议。

### Q6：这个 Demo 解决了什么具体痛点？

它解决的是实验研发链路的组织问题：

1. 自然语言目标难以直接执行；
2. 实验变量和控制变量容易混乱；
3. 批量实验日志难以复盘；
4. 失败原因常常只停留在主观描述；
5. 下一轮实验建议缺少结构化依据。

系统把这些步骤串起来，形成一个可执行闭环。

### Q7：它不是普通仿真 Demo 的地方在哪里？

普通仿真 Demo 往往是固定场景、固定动作、固定展示。我的项目额外做了：

- 自然语言任务解析；
- 机器人、物体、动作原语注册；
- 动态 MuJoCo 场景组装；
- 实验变量矩阵生成；
- 批量执行和结构化日志；
- 失败类型和指标统计；
- 下一轮实验建议；
- 前端实验控制台和报告生成。

网页只是入口，核心是后端实验闭环和 agent 编排。

## 3. 系统整体架构

### Q8：请讲一下系统架构。

系统可以分成五层：

```text
用户目标 / Web 控制台
        |
Agent Orchestrator
  - 任务解析
  - 实验设计
  - 结果分析
        |
Experiment Protocol Layer
  - TaskSpec
  - 变量矩阵
  - 成功标准
        |
Execution / Instrument Layer
  - MuJoCo runner
  - 已注册任务
  - 动作原语
  - IK solver
        |
Data Feedback Layer
  - logs
  - metrics
  - replay frames
  - reports
```

这个架构和智能实验室方向很接近：上层是研究意图，中间是实验协议，底层是仪器执行和数据回流。

### Q9：后端核心模块有哪些？

主要在 `mujoco_bridge/`：

- `agent.py`：智能体编排，解析任务、设计实验、分析结果；
- `server.py`：HTTP API 服务；
- `robot_registry.py`：解析和注册 8 个 MuJoCo Menagerie 机械臂；
- `scene_composer.py`：动态组装 MuJoCo XML 场景；
- `motion_primitives.py`：通用动作原语和 MotionPlan；
- `ik_solver.py`：Jacobian IK；
- `metrics_engine.py`：接触、位移、力等指标；
- `tasks/`：7 个已注册任务 runner；
- `arm_runner.py`：FR3 技能 runner。

### Q10：前端做了什么？

前端在 `web_demo/`，是一个实验控制台。它支持：

- 输入自然语言研发目标；
- 选择机器人和实验规模；
- 一键运行 NLP 管线；
- 选择传统任务 runner；
- 配置 MuJoCo API；
- 展示进度条；
- 展示实验设计、变量空间、指标卡片；
- 展示失败分布和变量影响；
- 展示成功/失败轨迹回放；
- 生成和下载 Markdown 报告。

### Q11：后端 API 有哪些？

主要 API：

- `GET /health`：健康检查；
- `GET /api/robots`：机器人列表；
- `GET /api/tasks`：任务列表；
- `GET /api/object_library`：物体库；
- `GET /api/asset_registry`：资产注册表；
- `GET /api/nlp/status`：NLP 管线进度；
- `POST /api/nlp/run`：NLP 智能管线；
- `POST /api/run_experiments`：传统批量实验；
- `POST /api/agent/run`：智能体闭环；
- `POST /api/arm/run_skill`：单个 FR3 技能。

## 4. 智能体到底在哪里

### Q12：你的“智能体”具体在哪里？

智能体主要在高层，不在低层实时控制。它负责：

1. 理解自然语言目标；
2. 选择任务或机器人；
3. 生成实验变量空间；
4. 组织 MuJoCo 实验；
5. 分析结构化结果；
6. 给出下一轮实验建议。

低层动作执行由 MuJoCo、IK、MotionPrimitive 和 runner 完成。这个设计更安全、更可验证。

### Q13：这和普通 pipeline 有什么区别？

普通 pipeline 是固定输入、固定流程、固定输出。这里的 agent 会根据不同研究目标动态决定任务、变量、动作序列和分析重点。同时它能把实验结果反馈到下一轮实验建议中。

当然，当前版本还不是完全自主科研系统。它是一个 constrained agent workflow：智能体在已注册工具、动作和实验协议边界内进行决策。

### Q14：LLM 做了哪些事情？

模型可用时，LLM 做三件事：

1. `parse_task_description`：把自然语言目标解析成 TaskSpec；
2. `design_experiment`：生成变量空间、假设和实验设计；
3. `analyze_runs`：根据实验日志和 summary 生成失败归因和下一轮建议。

模型不可用时，本地规则 fallback 会保证 Demo 仍然可运行。

### Q15：如果 LLM 不可用，系统还能跑吗？

能。系统有本地 fallback：

- `_local_parse_task`：规则解析任务；
- `_local_design_experiment`：规则生成变量空间；
- `_local_analyze_runs`：规则分析结果。

fallback 智能性较弱，但能保证演示稳定，不会因为模型 API 挂掉而无法运行。

### Q16：如果 LLM 输出不存在的机器人、物体或动作怎么办？

系统有白名单和校验：

- robot_id 必须在 robot registry 中；
- object_id 必须在 object library 中；
- action type 必须属于允许集合；
- 实验变量会被裁剪到 runner 支持范围；
- 场景 XML 组装后必须能被 MuJoCo 加载。

LLM 输出不会直接执行，必须通过结构化校验层。

### Q17：为什么不让 LLM 直接实时控制机器人？

实时控制需要稳定、连续、安全和低延迟。LLM 是文本生成模型，不适合毫秒级控制，也难保证动力学稳定。我的设计是让 LLM 做实验员和调度员，低层控制交给 IK、运动原语和 MuJoCo。

这符合科研实验系统的安全原则：模型可以提计划，但执行必须可验证、可约束。

### Q18：如果面试官质疑“这只是规则系统，不是智能体”，怎么回答？

我会承认系统中有很多规则，但这不是缺点。真实科研智能体落地必须有工具、协议和安全边界。LLM 负责目标理解、实验设计和分析，规则负责执行约束和验证。二者结合，才是可执行、可验证的科研智能体工作流。

## 5. 两条执行管线

### Q19：NLP 智能管线是怎么跑的？

流程是：

```text
POST /api/nlp/run
  -> 解析自然语言任务
  -> 选择机器人
  -> 组装 MuJoCo 场景
  -> 构建 MotionPlan
  -> 生成实验矩阵
  -> 运行 MuJoCo 批量实验
  -> 汇总指标
  -> 分析失败和下一轮建议
  -> 渲染成功/失败轨迹
```

这是最能体现智能实验闭环的路径。

### Q20：传统任务管线是怎么跑的？

传统任务管线通过 `/api/run_experiments` 调用已注册任务 runner。它不一定需要 LLM，适合稳定演示和对照实验。当前有 7 个任务：

- `fr3_arm_primitives`
- `fr3_pick_place`
- `tabletop_push`
- `screwdriving`
- `tool_use`
- `assembly_insertion`
- `cloth_folding`

### Q21：`/api/agent/run` 和 `/api/nlp/run` 有什么区别？

`/api/agent/run` 更像“在已注册任务上做 agent 编排”：模型帮助路由任务、设计变量空间、分析结果，但执行通常落到固定 runner。

`/api/nlp/run` 更像“从自然语言动态生成实验”：它会解析 TaskSpec、动态组装场景、生成 MotionPlan，再执行实验。

简单说：`agent/run` 更稳定，`nlp/run` 更通用。

### Q22：什么时候应该走已注册 runner，什么时候走动态 NLP 管线？

如果任务命中已注册 runner，并且 runner 能覆盖目标，优先走 runner，因为稳定、可验证、成功判定更清楚。

如果任务比较新，但可以由通用动作原语表达，则走 NLP 动态管线。

如果既没有 runner，也无法由动作原语表达，就应该返回 capability gap，而不是强行执行。

## 6. 动作序列与机器人控制

### Q23：当前动作序列有哪些？

NLP 动态管线支持 8 个通用动作原语：

- `reach`
- `grasp`
- `lift`
- `place`
- `push`
- `insert`
- `release`
- `wait`

默认 pick-and-place 是：

```text
reach(pre_grasp)
-> reach(grasp_pos)
-> grasp()
-> lift()
-> reach(place_pos)
-> release()
-> wait()
```

### Q24：FR3 已封装技能有哪些？

当前 FR3 技能有 6 个：

- `reach_touch`：移动并接触物体；
- `button_press`：按压按钮目标；
- `pick_lift`：抓取并抬升；
- `contact_sweep`：接触后扫动物体；
- `tool_contact_sweep`：手持工具扫动目标；
- `peg_insert`：手持 peg 插入 socket。

### Q25：自然语言如何变成动作序列？

LLM 输出结构化 actions，例如：

```json
[
  {"type": "reach", "target_pos": [0.5, 0.0, 0.5]},
  {"type": "grasp", "force_scale": 1.0},
  {"type": "lift", "height": 0.08},
  {"type": "place", "target_pos": [0.5, 0.15, 0.38]}
]
```

后端用 `MotionPlan.from_action_sequence()` 把这些 action 转成 MotionPrimitive。如果没有 actions，就使用默认 pick-and-place plan。

### Q26：为什么动作原语这么少？

这是有意控制范围。项目的第一目标是验证实验闭环，而不是实现完整机器人技能库。少量动作原语能让系统可控、可验证、容易调试。后续可以扩展为分层技能库：

- 原子动作；
- 接触技能；
- 任务技能；
- 实验协议。

### Q27：IK 是怎么做的？

使用 Jacobian 阻尼最小二乘 IK：

```text
dq = J^T (J J^T + lambda^2 I)^-1 e
```

MuJoCo 提供 site/body Jacobian，系统根据末端误差求解关节增量，并进行关节限位和阻尼处理。它适合 reach、lift、place 这类局部运动。

### Q28：为什么不用全局运动规划？

因为当前项目目标不是复杂路径规划，而是实验闭环。Jacobian IK 足以支撑基础操作任务，工程上更轻量。未来如果要处理复杂避障、多物体动态场景，可以接 OMPL、MoveIt 或采样式规划器。

## 7. MuJoCo 与物理仿真

### Q29：MuJoCo 在系统中是什么角色？

MuJoCo 是可编程实验仪器。它接收实验配置，执行物理仿真，输出结构化观测。它不是只用来画动画，而是产生接触、位移、力、状态和轨迹数据。

### Q30：MuJoCo 数据怎么被读取？

系统从 MuJoCo `data` 中读取：

- body position；
- geom contact；
- contact count；
- force-related values；
- object displacement；
- lifted height；
- final object pose；
- end-effector pose。

这些数据进入 success 判定和 failure_type 分类。

### Q31：接触检测怎么做？

MuJoCo 每步会生成 `data.contact` 和 `data.ncon`。系统遍历接触对，判断末端、工具、目标物体之间是否接触，统计接触步数和最大接触强度。接触不足会导致 `no_contact`、`weak_contact`、`grasp_miss` 等失败类型。

### Q32：真实渲染帧怎么生成？

后端用 `mujoco.Renderer` 做离屏渲染，在执行循环中通过 render callback 采样帧。帧会编码成 JPEG/base64 或保存到 replay assets，前端 canvas 播放。这个回放来自真实 MuJoCo 渲染，不是 2D 示意图。

### Q33：MuJoCo 数值不稳定怎么办？

MuJoCo 可能出现 QACC NaN/Inf，这通常来自接触参数、质量、控制增益、时间步或约束设置问题。处理方式：

1. 限制摩擦、力、速度、质量范围；
2. 场景组装后先做 smoke test；
3. 捕获异常并标记为 `simulation_unstable`；
4. 保留触发不稳定的变量组合；
5. 使用更保守的控制参数降级重跑。

### Q34：如何证明不是假仿真？

我用项目自带验证脚本验证过：

- NLP 管线能完成任务解析、场景组装、运动规划、MuJoCo 执行和渲染帧生成；
- 7 个 task runner 能跑通；
- FR3 arm skill 能产生真实接触、位移、力和回放帧。

这说明 Demo 不只是前端展示，而是后端确实执行了 MuJoCo 仿真。

## 8. 机器人、物体和资产

### Q35：支持哪些机器人？

当前支持 8 个 MuJoCo Menagerie 机械臂：

- Franka FR3
- Franka Panda
- UR5e
- UR10e
- Kinova Gen3
- KUKA iiwa
- xArm7
- Lite6

`robot_registry.py` 会把不同机器人统一成 RobotSpec。

### Q36：RobotSpec 包含什么？

包含：

- robot_id；
- name；
- MJCF 路径；
- DOF；
- joint_names；
- actuator_names；
- actuator_type；
- end_effector_site；
- 是否有 gripper；
- gripper 类型；
- keyframe qpos/ctrl。

这让上层 agent 不需要理解每个 MJCF 的细节。

### Q37：物体库有哪些？

物体库包括基础几何体、扫描网格物体和 robosuite 物体，例如 cube、cylinder、sphere、screw_head、insertion_socket、screwdriver、hammer、spatula、dish_towel、round_nut、plate_with_hole 等。

### Q38：外部资产为什么这么大？

因为包含 MuJoCo Menagerie、robosuite、scanned objects 和 cloth folding 模板。完整资产接近 1GB，不适合直接完整上传 GitHub。工程化版本应该用 manifest + downloader 或 submodule 管理。

## 9. 成功率、失败类型和指标

### Q39：成功率是怎么计算的？

每个 run 会产生 `success: true/false`。成功率就是：

```text
success_count / num_runs
```

它是当前实验矩阵和当前成功判定下的结果，不代表策略泛化能力。

### Q40：成功率 100% 是什么意思？

表示这轮实验里所有 run 都满足当前 success 阈值。它不代表机器人策略完美，也不代表训练完成。可能只是变量空间较容易、阈值偏宽、样本数较少，或者部分弱成功也被计为成功。

### Q41：如果面试官质疑 100% 没有区分度，怎么答？

可以答：

> 是的，如果一轮实验全是 100%，说明当前变量空间对任务区分度不够，或者成功判定偏宽松。这个结果应该触发下一轮主动实验设计：扩大扰动范围、提高成功标准、增加困难变量，或者把 strict success 和 weak success 分开统计。我的目标不是展示固定高成功率，而是找到失败边界。

### Q42：有哪些失败类型？

不同任务不同，常见有：

- `no_contact`
- `weak_contact`
- `grasp_miss`
- `lift_failed`
- `weak_displacement`
- `slip`
- `collision`
- `overshoot`
- `undershoot`
- `misalignment`
- `jammed`
- `not_inserted`
- `no_tool_contact`
- `simulation_unstable`

失败类型主要由物理指标和 runner 规则判定，不是 LLM 主观判断。

### Q43：为什么 weak success 也可能算 success？

当前系统为了区分“完全失败”和“部分完成”，有些弱完成会被标记为 success，同时保留 failure_type 或弱成功标记。这适合 Demo，但正式评估时应该拆成：

- strict success；
- partial success；
- contact success；
- task progress score。

### Q44：如何改进指标体系？

我会加入：

1. 严格成功率；
2. 部分成功率；
3. 接触率；
4. 平均物体位移；
5. 抬升高度分布；
6. 失败类型分布；
7. 变量水平置信区间；
8. 显著性检验。

## 10. 实验设计与统计可信度

### Q45：实验变量如何选择？

变量要满足三点：

1. 物理上可控；
2. 与失败模式相关；
3. 当前 runner 能执行和观测。

例如抓取任务中，摩擦、物体位置、抓取高度就是好变量。

### Q46：当前实验设计有什么不足？

当前主要是矩阵扫描，统计分析较粗。它能做初步变量影响分析，但还没有主动采样、显著性检验、置信区间和多因素交互建模。

### Q47：如何让失败归因更可信？

需要把 LLM 分析和统计证据结合：

- 变量分组成功率；
- 失败类型分布；
- 置信区间；
- 卡方检验/Fisher 检验；
- logistic regression；
- 多因素交互项；
- 重复实验和随机种子。

### Q48：下一轮实验建议如何生成？

当前模型可用时由 LLM 基于结构化日志生成；模型不可用时由本地规则生成。未来应该结合主动实验设计：根据不确定性和失败边界选择最有信息量的下一批实验。

### Q49：如何控制实验成本？

可以分阶段：

1. 粗粒度扫描找敏感变量；
2. 固定不敏感变量；
3. 在失败边界附近加密采样；
4. 用主动学习或 Bayesian optimization 选择实验点；
5. 对高成本实验先做小样本验证。

## 11. Demo 操作与演示路径

### Q50：面试现场应该演示哪条路径？

推荐演示 NLP 智能管线：

1. 启动后端；
2. 打开 `web_demo/index.html`；
3. 输入自然语言目标；

7. 展示指标、失败分析、报告和回放。

如果担心现场不稳定，可以用已注册 runner 或预先准备的结果。

### Q51：如果现场 MuJoCo 跑慢怎么办？

可以选择小规模实验，比如 1、9 或 27 组；也可以使用 fallback runner 或预渲染 replay。解释时强调：完整系统支持真实 MuJoCo，现场为了时间使用小规模演示。

### Q52：如果模型 API 不通怎么办？

系统会 fallback 到本地解析和分析，仍能跑闭环。可以主动说明这是工程上为了稳定演示做的设计。

1. 是否用 `.venv\Scripts\python.exe -m mujoco_bridge.server` 启动；
2. API 是否是 `http://127.0.0.1:8765/api/run_experiments`；
3. `/health` 是否返回 ok；
4. 浏览器是否允许本地文件访问本地 API；
5. CORS 是否正常。

## 12. API、前端和进度

### Q54：前端如何获取进度？

前端运行 NLP 管线时轮询 `GET /api/nlp/status`。后端在任务解析、机器人选择、场景组装、运动规划、实验执行、分析等阶段更新进度。

### Q55：你遇到过什么前端/后端同步问题？

有一个典型问题：进度状态最开始写在 handler 实例属性里，前端轮询请求可能访问另一个实例，所以只能看到 0% 和 100%。后来改成类属性，让所有请求共享状态。

### Q56：为什么不用 WebSocket？

本地单用户 Demo 用轮询更简单稳定。真正部署成多人实验平台时，WebSocket 或 SSE 更合适，可以推送日志、进度和中间结果。

### Q57：报告是怎么生成的？

前端根据当前 runs、summary、findings 和 recommendations 生成 Markdown 报告，可以下载。它是实验记录的一部分，体现数据回流和可复盘。

## 13. 安全、脱敏和 GitHub

### Q58：如何避免 API key 泄露？

真实 key 在 `configs/model_api.local.json`，该文件应加入 `.gitignore`。对外只提交 `model_api.local.example.json`。上传前必须检查 git status 和 staged files。

### Q59：文档如何脱敏？

脱敏包括：

- 去掉本机绝对路径；
- 去掉用户名；
- 去掉 API key；
- 去掉真实账号、token、私有数据；
- 只保留相对目录和示例配置。

### Q60：为什么 GitHub 不能直接上传完整 external？

因为外部资产接近 1GB，GitHub 仓库会非常重，也不利于评审快速查看。更合理的是保留核心代码和少量演示资产，大资产通过下载脚本、submodule 或 release 管理。

### Q61：如何发链接给别人演示？

两种方式：

1. 静态前端链接：GitHub Pages/Vercel 部署 `web_demo/`，展示样例数据和预渲染结果；
2. 完整在线 Demo：额外部署后端 MuJoCo API，前端连接公网 API。

前者稳定，后者真实但工程成本高。

## 14. 科研智能体与智能实验室相关问题

### Q62：如何类比 BohrClaw？

BohrClaw 这类科研智能体负责长期科研任务：理解目标、调用文献/模型/工具/实验、回收结果并推进下一步。我的项目是这个思想在机器人操作实验中的一个垂直切片：理解机器人实验目标，调用 MuJoCo 实验工具，回收结构化结果，推进下一轮实验。

### Q63：你的系统可以作为大科研智能体中的哪个模块？

可以作为 experimental execution tool。对外暴露：

- `list_capabilities()`
- `propose_experiment(goal)`
- `run_experiment(protocol)`
- `get_results(run_id)`
- `analyze_results(run_id)`

上层科研智能体可以把它当成机器人实验工具调用。

### Q64：智能实验室最重要的抽象是什么？

我认为是 protocol 和 feedback。Protocol 定义实验如何执行，feedback 定义实验结果如何回流。没有 protocol，实验不可编程；没有 feedback，智能体无法迭代。

我的项目中 TaskSpec 和 experiment matrix 对应 protocol，runs、metrics、summary 和 report 对应 feedback。

## 15. 项目边界和不足

### Q65：当前项目最大的边界是什么？

主要有：

1. 不是强化学习系统；
2. 不接真实机器人硬件；
3. 动作原语有限；
4. 运动规划不含复杂避障；
5. 成功判定是工程阈值；
6. 统计分析还不够严谨；
7. 部署和资产管理还未工程化。

### Q66：如果继续做一个月，优先做什么？

我会做：

1. strict/partial success 分层；
2. 变量影响统计检验；
3. 实验历史驱动的下一轮变量收窄；
4. capability gap 检测；
5. GitHub 轻量提交版；
6. 静态前端 Demo 链接；
7. 后端异步 job queue。

### Q67：如果接真实机器人，最先改什么？

改 runner 层，把 MuJoCo runner 替换成真实设备 adapter。接口可以是：

```text
prepare(protocol)
run(config)
collect()
shutdown()
```

上层 agent、实验矩阵、报告和分析尽量复用。

### Q68：如果任务超出能力范围，系统应该怎么做？

应该返回 capability gap，而不是假装执行。比如液体倒液任务需要液体仿真和容器模型，当前系统不支持，就应说明缺少 object、physics 和 action primitive，并建议补充能力。

## 16. 代码级追问

### Q69：`TaskSpec` 里应该有哪些字段？

包括：

- task_id/task_type；
- title/description；
- keywords；
- robot_candidates；
- objects；
- actions；
- workspace；
- experiment_variables；
- success_criteria；
- metrics；
- failure_types；
- supported_objects；
- execution_kind；
- runner_status。

### Q70：`MotionPlan.from_action_sequence()` 做什么？

它把 LLM 或本地解析器输出的 actions 转换为 MotionPrimitive 序列。例如 action type 是 `reach` 就生成 `MotionPrimitive.reach()`，是 `grasp` 就生成 `MotionPrimitive.grasp()`。

### Q71：`scene_composer.py` 的难点是什么？

难点是把不同来源的 MJCF 资产组合成一个有效场景，包括机器人、物体、工作台、工具、asset、actuator、tendon、default 等，还要避免命名冲突，并保证最终 XML 能被 MuJoCo 加载。

### Q72：`robot_registry.py` 的价值是什么？

它把不同机器人模型统一成 RobotSpec。否则每个机器人 MJCF 的 joint、site、actuator、gripper 定义都不同，上层代码会非常混乱。

### Q73：`metrics_engine.py` 的价值是什么？

它把物理观测和任务成功判定分离出来，避免每个 runner 都随意写指标逻辑。未来接真实仪器时，也可以把仪器数据转成统一 metrics。

### Q74：`experiment_history.py` 的价值是什么？

它用于多轮实验记忆，记录历史变量空间、结果和成功率变化。后续可以基于它做变量空间收窄、收敛检测和长期实验规划。

## 17. 面试陷阱问题

### Q75：你这个是不是已经实现了自主科学发现？

不能这么夸大。当前是科研实验闭环原型，能组织和执行实验，并给出下一轮建议，但还没有完整自主科学发现能力。它缺少长期知识库、严格统计推断、跨论文知识整合和真实实验仪器接入。

### Q76：你是不是训练了一个机器人策略？

没有。当前没有训练 RL 或 imitation learning 策略。机器人动作由脚本化运动原语、IK 和 runner 执行。项目重点是实验设计与执行闭环，不是策略学习。

### Q77：是不是所有自然语言任务都能做？

不是。系统只能处理已有机器人、物体和动作原语能表达的任务。超出能力范围的任务应该返回 capability gap。

### Q78：成功率高是不是说明你的策略强？

不一定。成功率必须结合变量空间、任务难度和成功阈值解释。高成功率可能说明任务容易或阈值宽。更重要的是通过主动实验找到失败边界。

### Q79：LLM 会不会产生幻觉？

会，所以系统不信任裸输出。它用白名单、schema、后端校验、MuJoCo 加载验证和 runner 支持空间裁剪来约束 LLM。

### Q80：这个系统和自动化脚本有什么不同？

自动化脚本执行固定流程；这个系统把研究目标解析、变量设计、执行、数据回流和下一轮建议连接起来。它有 agent 编排和实验反馈闭环，而不是单次脚本执行。

## 18. 薄弱点专项追问

### Q81：当前项目最大的薄弱点是什么？

最核心的薄弱点不是单个功能，而是“闭环深度”还需要继续加强。当前系统已经完成自然语言目标、TaskSpec、ExperimentPlan、SkillPlan、MuJoCo 执行、EvaluationReport、RetryPlan，并支持自动执行一轮修正实验；但它还没有扩展成长期、多轮、带收敛判断的自主实验循环。

所以当前更准确的定位是实验工作流 Agent 原型，而不是完全自主实验 Agent。下一步要补的是多轮 retry executor：把一次重试扩展成“多轮参数修改、自动重跑、对比前后实验结果、判断是否收敛”的循环。

### Q82：如果面试官问“你这个还不是真正闭环”，怎么答？

可以答：

> 是的，严格来说当前是最小闭环，而不是完全长期自主闭环。它已经能把 EvaluationReport 转成 RetryPlan，并自动执行一轮修正实验，输出前后对比；下一步是把一次 retry 扩展成多轮执行队列，让系统能持续完成“失败 -> 修改参数 -> 重跑 -> 对比 -> 判断是否继续”的循环。

这个回答的关键是承认边界，同时说明已经有结构化入口。

### Q83：如何把 RetryPlan 从建议变成自动执行？

需要增加一个 retry executor：

```text
EvaluationReport
  -> RetryPlan
  -> RevisedTaskSpec / RevisedExperimentPlan
  -> ExperimentMatrix
  -> MuJoCo execution
  -> ComparativeEvaluationReport
```

具体策略：

- `undershoot`：增加 push_force 或延长距离。
- `overshoot`：降低 push_force 或缩短距离。
- `no_contact`：调整接触点、接近高度或目标位置。
- `table_penetration`：提高接近高度、启用更严格桌面 clearance。
- `misalignment`：加入 align skill 或降低插入速度。

验收方式是展示同一个任务第一次失败、第二轮参数变化、第二轮指标改善。

### Q84：当前运动规划为什么是薄弱点？

当前主要是动作原语 + Jacobian IK，适合基础 reach、push、press、insert，但它不是完整运动规划器。它缺少：

- 全局路径搜索；
- 全身碰撞规避；
- 连续碰撞检测；
- 接触丰富任务的约束优化；
- 多阶段任务的长期规划。

面试时可以这样说：

> 当前阶段选择 IK 和动作原语，是为了优先验证实验智能体闭环。真正产品化时，底层 MotionPlan 应该替换或扩展为 MoveIt/OMPL/TrajOpt 这类规划器，上层 TaskSpec、SkillPlan 和 EvaluationReport 可以保持不变。

### Q85：机械臂陷进桌子的问题怎么解释？

这是非常关键的追问。

之前的问题本质是成功判定和物理安全没有统一：只看物体位移，容易把不合理运动判成功。现在已经补了：

- 物体初始高度按几何尺寸放到桌面上方；
- 成功条件中加入 `target_displacement_m`；
- 执行日志记录 `table_clearance_ok`；
- 记录 `min_ee_z`、`table_contact_steps`、`table_contact_pairs`；
- 只要桌面安全失败，就归因为 `table_penetration`，不能算成功。

但也要承认：

> 当前只是桌面任务的安全补强，不等价于完整全身碰撞规划。后续还需要机器人全身碰撞模型、连续碰撞检测和约束轨迹优化。

### Q86：为什么不现在就接 MoveIt 或 OMPL？

回答重点：

> 这个 Demo 的目标优先级是验证“实验智能体闭环”，不是做完整机器人规划栈。直接接 MoveIt 会增加大量工程依赖，反而掩盖 TaskSpec、实验设计、数据回流和失败归因这些核心贡献。当前用轻量 IK 保持系统可控；后续可以把 MotionPlan 层替换为 MoveIt/OMPL，而不影响上层智能体协议。

也就是说，不是不会接，而是当前阶段有意保持执行层轻量。

### Q87：如果真实硬件接入，最大的风险是什么？

最大风险是安全和 sim-to-real gap。

仿真里成立的动作，在真实环境中可能因为标定误差、摩擦差异、传感噪声、控制延迟和碰撞风险失败。真实接入必须先做：

- 工作空间边界检查；
- 速度、加速度、力限制；
- 碰撞预检查；
- 急停状态读取；
- 人工确认；
- 真实执行日志；
- 低风险任务白名单。

当前 RealRobotAdapter 是 mock 安全门控，说明架构上已经为真实硬件预留接口，但没有贸然直接控制真实机械臂。

### Q88：如何从 MuJoCo 迁移到真实机器人？

核心是保持上层协议不变，只替换执行适配器：

```text
TaskSpec / SkillPlan
  -> SimAdapter(MuJoCo)
  或
  -> RealRobotAdapter(真实机械臂 SDK)
```

真实机器人侧需要把 SkillPlan 翻译成具体控制器指令，例如 ROS action、厂商 SDK 或实时控制接口。执行后再把真实传感器数据转换为统一 ExecutionTrace 和 EvaluationReport。

这就是为什么系统里要有 Adapter 抽象。

### Q89：实验记忆为什么还是薄弱点？

当前系统可以输出单轮实验日志和报告，但还没有充分把历史实验变成下一轮规划依据。

真正的实验智能体应该能回答：

- 这个任务以前跑过吗？
- 哪组参数成功率最高？
- 最常见失败原因是什么？
- 哪些变量已经证明不敏感？
- 下一轮应该复用还是避开哪些参数？

补强方式是建立 ExperimentStore，把 TaskSpec、ExperimentPlan、runs、EvaluationReport、RetryPlan 都写入结构化存储，再让下一轮实验设计读取历史记录。

### Q90：如何让系统从规则变得更“智能”？

不应该直接把规则全部替换成 LLM，而是让智能体逐步在三个位置增强：

1. 任务理解：从模板解析升级到 schema-guided parsing 和 task ontology。
2. 实验设计：从固定矩阵升级到主动采样、失败边界搜索、Bayesian optimization。
3. 经验复用：从单轮报告升级到历史实验检索和参数推荐。

底层安全、能力边界和执行校验仍然应该保留规则。

### Q91：如果面试官说“这就是规则 + LLM 包装”，怎么答？

推荐回答：

> 真实可执行 Agent 本来就不应该是裸 LLM。它必须由 LLM、工具、协议、约束和验证层组成。这个项目的 agentic 部分体现在它能从目标生成实验协议，调用 MuJoCo 工具，读取结构化反馈，并生成下一轮实验建议；规则和 schema 是为了保证执行安全和可复现，不是为了削弱智能体。

这个回答要强调：可控边界是工程优势。

### Q92：统计可信度不足怎么补？

当前 Demo 的成功率是小样本、工程阈值下的结果。正式评估要补：

- 样本数和成功次数；
- 置信区间；
- strict success / partial success 分层；
- 变量分组成功率；
- 显著性检验；
- 多因素交互分析；
- 随机种子和重复实验；
- 低样本提醒。

面试口径：

> 当前结果证明链路可运行，不声称跨场景泛化。后续会把 EvaluationReport 从 demo report 升级为统计报告。

### Q93：如果现场 Demo 跑出失败怎么办？

不要慌，失败也是展示点。

正确讲法：

> 这个系统不是为了隐藏失败，而是为了把失败结构化。失败时我会看 failure_type、位移误差、接触步数和 table_clearance_ok，然后解释它属于 no_contact、undershoot、overshoot 还是 table_penetration，并展示 RetryPlan 如何给出下一轮调整。

这反而更符合实验智能体的定位。

### Q94：如何量化下一阶段是否真的进步？

可以设四个验收指标：

- 自动重试：至少 3 类失败能自动生成第二轮计划并重跑。
- 规划安全：桌面任务中所有成功样本必须满足 `table_clearance_ok=true`。
- 统计报告：每个成功率都显示样本数和置信提示。
- 记忆复用：同类任务第二次运行能读取历史最佳参数或历史失败原因。

这比泛泛说“继续优化”更有工程可执行性。

### Q95：哪些薄弱点可以主动讲，哪些不要主动放大？

可以主动讲：

- 当前不是端到端控制 Agent；
- RetryPlan 已能自动执行一轮，但还没有形成长期多轮自主闭环；
- 运动规划还不是完整全局规划；
- 仿真结果不等于真实硬件结论；
- 小样本成功率不能代表泛化。

不要主动放大成：

- “这个系统还很不成熟”；
- “目前只是规则脚本”；
- “还没有什么智能”。

更好的表达是：

> 当前是最小闭环原型，核心架构已经打通；薄弱点集中在闭环深度、运动规划、真实硬件和长期记忆，这些都能沿着现有协议化架构继续补齐。

### Q96：如果只能用一句话总结薄弱点和补强方案？

可以说：

> 当前项目已经完成实验工作流 Agent 的最小闭环，并支持一次自动重试；薄弱点主要是还没有长期多轮 retry、底层规划仍是轻量 IK、真实硬件还未接入和长期实验记忆不足。下一步会沿着 TaskSpec、SkillPlan、Adapter、EvaluationReport 四个接口，把它从“一次闭环的实验 Agent”升级为“会多轮自主推进的实验 Agent”。

## 19. 可以直接背的总结回答

### 30 秒版

我这个 Demo 是一个面向机器人操作实验的智能实验闭环系统。它把自然语言研究目标转成结构化实验协议，调用 MuJoCo 作为可编程实验仪器批量执行，回收物理指标和失败类型，再由智能体分析结果并提出下一轮实验建议。它的价值不在于训练一个最优机器人策略，而在于验证“目标 - 实验设计 - 执行 - 数据回流 - 决策迭代”的科研智能体链路。

### 2 分钟版

这个项目可以理解成一个小型 AI 原生实验操作系统。用户输入研究目标，比如研究摩擦和位姿扰动对抓取成功率的影响。系统会解析任务，选择机器人和物体，生成动作序列和实验变量空间，然后通过 MuJoCo 执行批量实验，记录成功率、接触步数、物体位移、抬升高度、最大接触力和失败类型。最后，智能体基于结构化结果进行失败归因，并提出下一轮实验建议。

我把系统分成两条管线：NLP 动态管线负责从自然语言动态生成实验；传统任务管线负责稳定执行已注册任务。智能体主要在高层做实验设计和分析，低层执行由 MuJoCo、IK 和动作原语完成。这样既有智能体的灵活性，又保留了物理执行的可验证性。

### 如果被问和岗位最相关的一句话

这个项目和智能实验室方向的连接在于：我把 MuJoCo 抽象成可编程实验仪器，把 runner 抽象成实验协议，把 metrics/logs 抽象成数据回流，把 agent 抽象成科研工作流编排器。虽然当前是机器人实验，但这套“可编程执行 + 可验证反馈 + 智能决策”的闭环可以迁移到更广泛的科学实验自动化场景。

# 通用机械臂实验智能体建设路线图

## 0. 文档目标

本文档用于规划当前 Demo 从“可演示的机械臂实验闭环原型”演进为“真正可泛化、可验证、可扩展的通用机械臂实验智能体”的执行路径。

当前项目已经验证了一个基础闭环：

> 自然语言实验目标 -> 结构化任务表示 -> 动作序列生成 -> MuJoCo 仿真实验执行 -> 指标采集 -> 成败分析与报告输出

下一阶段的目标不是简单增加几个任务，而是把系统从“Demo 型流程”推进到“通用实验智能体平台”：

> 面向不同机械臂、不同工具、不同物体、不同实验任务，智能体能够自主理解目标、规划实验、调用仿真或真实设备、采集证据、分析失败、迭代策略，并形成可复现的实验记录。

---

## 1. 最终系统定义

### 1.1 一句话目标

构建一个面向机器人研发实验的通用机械臂实验智能体，使用户能够用自然语言描述实验目标，系统自动完成任务建模、机器人与工具选择、实验方案生成、仿真或真实执行、数据回流、失败诊断和下一轮实验建议。

### 1.2 最终系统应具备的核心能力

1. **任务理解能力**
   - 能理解自然语言实验目标。
   - 能识别任务类型、目标物体、工具、约束条件、成功标准。
   - 能把模糊目标转成结构化实验协议。

2. **实验规划能力**
   - 能根据任务目标生成动作序列。
   - 能选择合适的机械臂、末端执行器、工具和场景参数。
   - 能为不确定任务生成多组实验变量。

3. **执行控制能力**
   - 能在 MuJoCo 中完成物理仿真实验。
   - 能通过统一接口迁移到真实机械臂或远程实验设备。
   - 能处理执行失败、碰撞、不可达、夹取失败、插入失败等异常。

4. **数据回流能力**
   - 能采集轨迹、接触、力、位置、姿态、状态变化和任务结果。
   - 能保存实验配置、执行日志、视频或关键帧。
   - 能把一次实验变成可复查、可比较、可统计的数据记录。

5. **失败诊断能力**
   - 能判断失败属于感知问题、规划问题、控制问题、物理约束问题还是任务定义问题。
   - 能给出下一轮改进建议。
   - 能基于历史实验逐步优化参数和策略。

6. **泛化能力**
   - 能支持不同机器人模型。
   - 能支持不同物体与工具。
   - 能支持不同任务族：抓取、搬运、按压、插入、旋拧、推拉、装配、工具使用等。

7. **可信验证能力**
   - 能给出成功率、失败分布、置信度和实验覆盖范围。
   - 能说明当前结论适用于哪些条件，不适用于哪些条件。
   - 能形成面向科研或工程评审的实验报告。

## 2. 当前项目状态

### 2.1 已经具备的能力

当前 Demo 已经完成以下基础能力：

1. **自然语言到任务结构化**
   - 支持从自然语言中解析任务目标。
   - 能生成 TaskSpec 类结构化任务描述。
   - 能覆盖按压、插入、旋拧、工具接触、抓取搬运等典型任务意图。

2. **动作原语执行**
   - 已具备 reach、grasp、lift、place、push、insert、release、wait 等基础动作。
   - FR3 相关任务中已有 reach_touch、button_press、pick_lift、contact_sweep、tool_contact_sweep、peg_insert 等技能。

3. **MuJoCo 仿真实验执行**
   - 能动态组装场景。
   - 能加载不同机器人模型。
   - 能执行动作序列并获得物理反馈。

4. **实验指标与结果判断**
   - 能采集接触、位置、对象状态、成功条件等结果。
   - 能生成结构化结果。
   - 能支持批量实验和基础成功率统计。

5. **前端展示与报告输出**
   - 已具备基础 Web Demo。
   - 支持任务输入、执行进度、指标展示、回放和报告下载。

### 2.2 当前系统的本质定位

当前系统不是完整的通用机器人智能体，而是一个“机械臂实验智能体的最小可行原型”。

它已经证明了：

> 大模型或规则 Agent 可以作为实验工作流的大脑，把自然语言目标转化为可执行的机械臂实验，并通过 MuJoCo 获得可验证反馈。

但它还没有完全解决：

> 如何面对未知物体、未知任务、复杂约束、真实硬件、不确定执行、多轮自主实验和高可信统计验证。

## 3. 距离真正通用机械臂实验智能体的关键差距

### 3.1 任务泛化差距

当前问题：

- 支持的任务类型仍然有限。
- 任务理解主要围绕已知模板和少量规则。
- 对组合任务、长程任务、带约束任务支持不足。

需要补齐：

- 建立任务本体：任务类型、对象类型、工具类型、约束类型、成功标准。
- 支持复杂任务分解：把“完成一个装配实验”拆成多个可执行子任务。
- 支持多目标约束：例如“不能碰倒旁边物体”“夹取后保持水平”“插入深度达到 3cm”。

验收标准：

- 用户输入 30 类常见机械臂实验目标，系统能结构化解析其中至少 80%。
- 对每类任务能输出明确的任务类型、对象、工具、动作序列和成功条件。
- 对无法执行的任务能说明原因，而不是静默失败。

### 3.2 动作技能差距

当前问题：

- 动作原语偏基础。
- 技能之间缺少统一抽象。
- 复杂动作主要依赖预设逻辑，缺少可组合策略。

需要补齐：

- 建立统一 Skill 接口。
- 把抓取、放置、按压、插入、旋拧、推、拉、拨动、扫动、工具使用等能力模块化。
- 每个技能都要有输入、输出、前置条件、失败类型和可调参数。

验收标准：

- 至少实现 12 个可复用技能。
- 每个技能支持独立测试。
- 任意任务计划都能追溯到具体技能调用序列。

### 3.3 机器人泛化差距

当前问题：

- 当前虽有机器人注册表，但不同机器人之间的能力差异没有充分建模。
- 机械臂参数、工作空间、末端工具、关节限制、控制接口尚未抽象为统一能力描述。

需要补齐：

- 建立 RobotCapabilitySchema。
- 描述每台机器人支持的自由度、工作空间、末端执行器、最大负载、可用技能和控制方式。
- 智能体根据任务自动选择机器人，而不是只做静态切换。

验收标准：

- 至少支持 3 类机械臂模型的统一调度。
- 系统能根据任务目标自动判断某机械臂是否可执行。
- 对不可达目标能输出“机器人能力不足”或“目标超出工作空间”等原因。

### 3.4 场景与物体泛化差距

当前问题：

- 物体类别和几何属性有限。
- 场景生成仍偏工程脚本。
- 缺少可复用的对象库、工具库和实验台配置库。

需要补齐：

- 建立 ObjectLibrary、ToolLibrary、SceneTemplateLibrary。
- 每个对象记录尺寸、质量、摩擦、可抓取点、功能区域和任务相关属性。
- 场景配置可由任务自动生成。

验收标准：

- 至少支持 20 个常见实验对象。
- 至少支持 8 类工具或末端交互对象。
- 场景配置可以通过 JSON/YAML 复现。

### 3.5 规划能力差距

当前问题：

- 目前主要是模板化动作序列和局部 IK。
- 缺少任务级规划、约束规划和失败后的重新规划。
- 对复杂动作的搜索和优化能力有限。

需要补齐：

- 建立分层规划：
  - 任务层：目标拆解。
  - 技能层：选择技能组合。
  - 运动层：生成可执行轨迹。
  - 反馈层：根据执行结果调整。
- 引入规划器或策略搜索模块。
- 支持失败重试和参数自适应。

验收标准：

- 同一个任务失败后，系统能自动修改至少一种参数并重新执行。
- 能记录“第一次失败 -> 调整策略 -> 第二次成功”的完整闭环。
- 对失败原因能给出可验证解释。

### 3.6 实验设计差距

当前问题：

- 批量实验目前偏固定参数组合。
- 智能体尚未真正主动提出实验变量。
- 缺少假设、变量、指标、结论之间的明确结构。

需要补齐：

- 引入 ExperimentPlan。
- 每次实验都包含：
  - hypothesis：实验假设
  - variables：实验变量
  - controls：控制条件
  - metrics：评价指标
  - runs：执行次数
  - expected_result：预期结果
- 智能体能根据失败结果提出下一轮实验计划。

验收标准：

- 对同一任务，系统能自动生成至少 3 组有意义的变量配置。
- 能说明每组变量要验证什么。
- 能根据统计结果给出下一步实验建议。

### 3.7 数据与记忆差距

当前问题：

- 单轮实验数据已有，但长期实验记忆不足。
- 缺少跨实验比较、检索和经验积累。
- Agent 不能充分利用历史结果。

需要补齐：

- 建立 ExperimentStore。
- 每次实验保存：
  - 任务描述
  - 场景配置
  - 机器人配置
  - 动作序列
  - 执行日志
  - 指标结果
  - 失败原因
  - 视频或关键帧
  - Agent 分析结论
- 支持按任务、对象、机器人、失败类型检索历史实验。

验收标准：

- 同类任务第二次执行时，系统能读取历史成功参数。
- 报告中能引用历史实验结果。
- 能输出“历史上该任务最常见失败原因”。

### 3.8 真实硬件差距

当前问题：

- 当前以 MuJoCo 仿真为主。
- 还没有真实机械臂控制接口。
- 缺少仿真到真实的安全校验层。

需要补齐：

- 建立 InstrumentAdapter 抽象。
- 支持 SimAdapter 和 RealRobotAdapter。
- 加入真实执行前的安全检查：
  - 工作空间检查
  - 碰撞风险检查
  - 力/速度限制
  - 急停接口
  - 人工确认机制

验收标准：

- 同一任务协议可以在 SimAdapter 和 RealRobotAdapter 之间切换。
- 真实执行前能生成安全检查报告。
- 危险动作默认禁止直接执行。

### 3.9 可信评估差距

当前问题：

- 成功率统计已经有基础，但可信度表达不足。
- 缺少置信区间、样本量说明、条件覆盖说明。
- 报告还不够科研化。

需要补齐：

- 建立 EvaluationReport。
- 输出成功率、失败类型分布、样本数、置信区间、覆盖条件和边界条件。
- 区分“单次成功”“多次稳定成功”“跨场景泛化成功”。

验收标准：

- 报告明确区分单次实验结果和统计结论。
- 每个成功率都带样本数。
- 对低样本实验自动提示“不足以证明泛化能力”。

## 4. 总体技术路线

### 4.1 系统分层架构

目标系统建议采用 7 层架构：

1. **User Goal Layer**
   - 接收自然语言目标。
   - 支持用户补充约束和实验偏好。

2. **Task Understanding Layer**
   - 抽取任务类型、对象、工具、约束、成功标准。
   - 输出 TaskSpec。

3. **Experiment Planning Layer**
   - 根据 TaskSpec 生成 ExperimentPlan。
   - 设计变量、对照组、执行次数和评价指标。

4. **Skill Planning Layer**
   - 将实验计划转成技能序列。
   - 输出 SkillPlan。

5. **Execution Layer**
   - 调用 MuJoCo 或真实机械臂。
   - 负责轨迹执行、状态采集、异常处理。

6. **Evaluation Layer**
   - 计算成功条件、指标、失败原因和统计结果。

7. **Memory and Iteration Layer**
   - 保存实验历史。
   - 生成下一轮实验建议。
   - 支持经验复用。

### 4.2 核心数据对象

后续应逐步稳定以下结构：

```yaml
TaskSpec:
  task_type: pick_place | press | insert | screw | push | tool_use | assembly
  objects: []
  tools: []
  robot_requirements: {}
  constraints: []
  success_criteria: []
  safety_limits: []

ExperimentPlan:
  hypothesis: string
  variables: []
  controls: []
  metrics: []
  run_count: integer
  expected_result: string

SkillPlan:
  robot_id: string
  end_effector: string
  skills:
    - skill_name: string
      params: {}
      preconditions: []
      expected_observations: []

ExecutionTrace:
  timestamps: []
  joint_states: []
  object_states: []
  contacts: []
  forces: []
  events: []
  warnings: []

EvaluationReport:
  success: boolean
  success_rate: number
  sample_count: integer
  failure_reasons: []
  confidence_notes: []
  next_actions: []
```

## 5. 可执行阶段规划

周期建议：1 周

目标：

> 先让当前系统成为一个稳定、可复现、可解释的 Baseline，避免后续扩展建立在不稳定基础上。

- [ ] 所有失败都能归入明确类型。

## 9. 技术风险与应对方案

### 9.1 大模型输出不稳定

风险：

- TaskSpec 格式不稳定。
- 任务解析存在幻觉。
- 不可执行任务被错误执行。

应对：

- [ ] 使用 JSON Schema 校验。
- [ ] 对任务协议做二次验证。
- [ ] 对高风险任务强制人工确认。
- [ ] 保留规则解析 fallback。

### 9.2 仿真成功不等于真实成功

风险：

- MuJoCo 中可行的动作在真实机械臂上失败。
- 摩擦、接触、误差和传感器噪声没有充分建模。

应对：

- [ ] 明确区分仿真结论和真实结论。
- [ ] 加入噪声、扰动和随机初始条件。
- [ ] 增加 sim-to-real 安全检查。
- [ ] 真实执行前只允许低风险任务。

### 9.3 技能库过度工程化

风险：

- 过早抽象导致开发变慢。
- 技能接口复杂但实际任务不足。

应对：

- [ ] 先围绕当前任务抽象最小接口。
- [ ] 每新增一个技能必须有真实用例。
- [ ] 不为暂时不存在的任务做复杂设计。

### 9.4 规划器复杂度过高

风险：

- 一开始就做通用规划器会超出项目周期。

应对：

- [ ] 先做任务族内规划。
- [ ] 先支持失败重试和参数调整。
- [ ] 后续再引入更复杂的搜索或学习方法。

### 9.5 报告结论被误读

风险：

- 单次成功被误读为通用能力。
- 小样本成功率被误读为稳定性能。

应对：

- [ ] 报告中强制写样本数。
- [ ] 成功率必须绑定实验条件。
- [ ] 对低样本自动提示“仅代表 Demo 验证”。

## 11. 最终判断标准

当系统满足以下标准时，可以认为它已经接近“通用机械臂实验智能体”：

- [ ] 用户可以用自然语言提出新的机械臂实验目标。
- [ ] 系统能生成结构化实验协议。
- [ ] 系统能选择机器人、工具、对象和场景。
- [ ] 系统能把实验协议转成技能计划。
- [ ] 系统能在仿真中执行并采集证据。
- [ ] 系统能输出可信实验报告。
- [ ] 系统能解释失败原因。
- [ ] 系统能提出下一轮实验。
- [ ] 系统能复用历史经验。

> 一个以 Agent 为大脑、以机械臂和仿真器为实验执行层、以结构化数据和报告为反馈层的机器人实验操作系统原型。

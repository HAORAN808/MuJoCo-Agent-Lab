# 追光计划面试准备：实验工作流 Agent 项目叙事与技术问答

## 1. 项目主线

本项目是一个面向机器人操作实验的实验工作流 Agent 原型。项目目标不是展示一个固定机械臂动作，而是验证科研实验自动化中的关键闭环：

```text
自然语言实验目标
  -> 结构化实验协议
  -> 实验计划
  -> 技能计划
  -> 可编程实验执行
  -> 数据回流
  -> 严格评估
  -> 失败归因
  -> 自动重试
  -> 实验记忆
```

核心定位：

> 以机械臂实验为载体，构建一个可编程、可执行、可观测、可评估、可重试、可沉淀记忆的实验工作流 Agent。

## 2. 项目能力结构

| 能力层 | 本项目实现 |
| --- | --- |
| 目标理解 | 自然语言实验目标解析 |
| 实验协议 | TaskSpec |
| 实验设计 | ExperimentPlan |
| 技能规划 | SkillPlan / MotionPlan |
| 执行工具 | MuJoCo 虚拟实验仪器 |
| 结果回流 | Runs、Metrics、Replay |
| 可信评估 | EvaluationReport、目标误差、物理安全、样本数 |
| 失败处理 | RetryPlan、AutoRetry |
| 经验沉淀 | ExperimentStore |

项目完整闭环：

```text
科研目标
  -> 实验协议
  -> 设备/仿真执行
  -> 结果回流
  -> 失败归因
  -> 下一轮实验
```

## 3. 一分钟专业项目介绍

本项目是一个面向机器人操作实验的实验工作流 Agent 原型。用户输入自然语言实验目标，例如“用 FR3 机械臂推动方块移动 5cm，不能陷进桌子”，系统会将目标解析为结构化 TaskSpec，生成 ExperimentPlan 和 SkillPlan，完成机器人、工具和物体能力匹配，调用 MuJoCo 作为可编程虚拟实验仪器执行实验，并回收位移、接触、失败类型、轨迹回放和物理安全指标。

与普通机器人仿真不同，本项目强调实验闭环：执行后系统会生成 EvaluationReport，根据失败类型生成 RetryPlan，并自动执行一轮修正实验，最后将任务协议、实验结果和重试摘要写入 ExperimentStore。它验证的是“自然语言目标到可验证实验闭环”的 Agentic Workflow，而不是单个固定动作的动画展示。

## 4. 面试官最可能关注的能力

### 4.1 Agent 系统设计能力

重点问题：

- 为什么这算 Agent？
- Agent 和 pipeline 的边界是什么？
- Agent 是否直接控制机器人？
- 如何避免 LLM 幻觉？
- 上下文和状态如何管理？
- 工具调用、Skills、Memory 的关系是什么？

回答核心：

> 这是一个 constrained workflow agent。Agent 负责目标理解、协议生成、实验设计、工具调用、结果分析和重试决策；执行层由 MuJoCo、IK、SkillPlan 和安全约束负责。LLM 不直接控制关节，也不绕过 schema、能力匹配和物理执行层。

### 4.2 实验操作系统能力

重点问题：

- TaskSpec 为什么必要？
- ExperimentPlan 如何设计？
- SkillPlan 如何从任务生成？
- 如何保证实验可复现？
- 如何管理实验历史？
- 如何从仿真迁移到真实实验设备？

回答核心：

> TaskSpec 是自然语言和可执行实验之间的协议层；ExperimentPlan 描述变量、样本数和指标；SkillPlan 描述机器人技能序列；EvaluationReport 和 ExperimentStore 让每轮实验可回溯、可对比、可积累。

### 4.3 机器人与物理执行能力

重点问题：

- MuJoCo 在系统中是什么角色？
- 动作序列有哪些？
- IK 怎么做？
- 物体穿桌为什么之前会发生，现在怎么避免？
- 如何判断 5cm 是否真的达标？
- 为什么不用 MoveIt/OMPL？

回答核心：

> MuJoCo 被抽象为虚拟实验仪器。当前运动层是动作原语 + Jacobian IK，用于验证实验闭环；安全层加入了目标位移、容差、桌面 clearance、接触对记录，所有成功样本必须同时满足任务目标和物理安全。MoveIt/OMPL 是后续 MotionPlanner 层的增强方向。

### 4.4 工程化与可靠性

重点问题：

- 如果模型 API 不通怎么办？
- 如果现场 Demo 失败怎么办？
- 如何做验证？
- 如何脱敏？
- 如何部署给别人看？
- 如何区分 Demo 结论和泛化结论？

回答核心：

> 系统有本地 fallback、验证脚本、严格约束回归、实验记忆和可复现报告。现场失败不是问题，因为项目重点是结构化失败和重试；不会把小样本成功率夸大成泛化能力。

## 5. 针对图片中 Agent 岗面试题的映射准备

### Q：Agent 平台多模块集成架构如何设计？

本项目采用分层架构：

```text
User Goal
  -> Agent Orchestrator
  -> TaskSpec / ExperimentPlan / SkillPlan
  -> Instrument Adapter
  -> MuJoCo Execution
  -> EvaluationReport
  -> RetryPlan / ExperimentStore
```

模块边界：

- Agent Orchestrator：解析、计划、分析。
- Protocol Layer：结构化任务协议。
- Skill Layer：动作和技能表达。
- Adapter Layer：仿真/真实设备执行接口。
- Evaluation Layer：指标、失败类型、成功率。
- Memory Layer：历史实验记录。

### Q：用户自然语言输入的具体形式是什么？

当前支持目标式输入，例如：

```text
用 fr3 机械臂推动方块移动5cm，不能陷进桌子
```

系统抽取：

- task_type: push
- robot: franka_fr3
- object: cube_5cm
- target_displacement_m: 0.05
- table clearance constraint
- actions: reach + push
- metrics: displacement_error、table_clearance_ok、failure_type

### Q：分析功能的输入、输出流程和核心目的是什么？

输入：

- TaskSpec
- runs
- summary
- failure_distribution
- physical metrics

输出：

- EvaluationReport
- failure_reasons
- RetryPlan
- AutoRetry comparison
- ExperimentStore record

核心目的：

> 把一次机械臂实验从“看起来成功/失败”变成可量化、可归因、可重试的实验记录。

### Q：用户输入灵活性如何实现？

当前采用两层机制：

- LLM 解析自然语言目标。
- 本地 fallback 规则保证无模型时也能运行。

解析结果必须经过：

- schema 化字段约束；
- robot/object/action 白名单；
- task_type 支持范围检查；
- MuJoCo 场景加载验证；
- success_criteria 后处理。

### Q：分析结果以什么结构化形式呈现？

主要结构：

```json
{
  "task_spec": {},
  "experiment_plan": {},
  "skill_plan": {},
  "runs": [],
  "evaluation_report": {},
  "retry_plan": {},
  "retry_execution": {},
  "experiment_memory": {}
}
```

页面展示对应：

```text
TaskSpec -> ExperimentPlan -> SkillPlan -> Capability -> MuJoCo -> Evaluation -> RetryPlan -> AutoRetry -> Memory
```

### Q：多 Agent 系统设计动机是什么？

当前系统没有拆成多个独立进程 Agent，但逻辑上已经有多角色分工：

- Parser Agent：目标解析。
- Planner Agent：实验计划和技能计划。
- Executor Tool：MuJoCo 执行。
- Evaluator：指标评估。
- Retry Agent：根据失败生成修正计划。
- Memory Manager：记录实验历史。

面试时不要强行吹“多 Agent 已经完整实现”，而是说：

> 当前是单 Orchestrator 下的多角色 agentic workflow，后续可以拆成多 Agent 协作系统。

### Q：模型选型策略是什么？

本项目不是模型能力竞赛，而是工具编排和实验闭环验证。模型选型原则：

- 任务解析需要稳定 JSON 输出。
- 分析归因需要长上下文和结构化推理。
- 高风险执行不能依赖裸模型输出。
- 模型不可用时必须 fallback。

因此模型层可以替换，核心价值在协议、工具、验证和反馈闭环。

### Q：上下文管理机制是什么？

当前上下文分三层：

- 任务上下文：goal、TaskSpec、SkillPlan。
- 执行上下文：runs、metrics、failure_type、replay。
- 记忆上下文：ExperimentStore 历史记录。

状态流转：

```text
goal
  -> task_spec
  -> experiment_plan / skill_plan
  -> runs
  -> evaluation_report
  -> retry_plan
  -> retry_execution
  -> experiment_memory
```

### Q：ReAct 模式如何应用？

当前不是严格论文式 ReAct prompt，但系统行为符合 ReAct 思路：

```text
Reason: 解析目标、判断任务、选择变量
Act: 调用 MuJoCo 执行实验
Observe: 读取 runs、metrics、failure_type
Reason: 失败归因、生成 RetryPlan
Act: 自动执行一轮修正实验
Observe: 对比重试前后结果
```

可以说：

> 我们没有把 ReAct 写成纯 prompt，而是把 Reason/Act/Observe 固化为工程模块，提升可验证性。

### Q：任务完成判定条件是什么？

推动任务示例：

```text
abs(object_displacement - target_displacement_m) <= tolerance
and table_clearance_ok == true
```

失败类型：

- no_contact
- undershoot
- overshoot
- table_penetration

任务完成不是 LLM 判断，而是后端根据物理指标判断。

### Q：为什么没有直接用 LangChain？

可以回答：

> 当前 Demo 的关键不是搭一个通用聊天 Agent，而是控制协议、物理执行、安全验收和实验记录。LangChain 可以用于模型调用和工具编排，但核心 TaskSpec、SkillPlan、MuJoCo Adapter、EvaluationReport 仍需要领域自定义。为减少依赖和提升可控性，当前先实现轻量 Orchestrator。

### Q：LangChain 与 LlamaIndex 怎么看？

回答：

- LangChain 强在工具编排、chain/agent 抽象、生态集成。
- LlamaIndex 强在数据索引、RAG、文档/知识库检索。
- 本项目核心不在文档 RAG，而在实验协议和执行闭环。
- 后续如果加入文献/历史实验检索，可以引入 LlamaIndex。
- 如果要对接更多工具和模型路由，可以引入 LangChain 或等价编排层。

### Q：Skills 颗粒度如何确定？

原则：

- 太细：Agent 难规划，执行链过长。
- 太粗：不可组合，不利于失败归因。

当前粒度：

```text
reach / grasp / lift / place / push / insert / release / wait
```

这是适合 Demo 的中等粒度。

后续可以升级为：

- atomic motion primitive
- contact skill
- task skill
- experiment protocol

### Q：MCP 标准协议如何落地？

当前未接 MCP，但架构可以自然扩展为 MCP 风格的工具与资源服务：

- MuJoCo Runner 可以封装为 tool server。
- ObjectLibrary、RobotRegistry 可以作为 resources。
- run_experiment、get_results 可以作为 tools。
- ExperimentStore 可以作为 memory/resource。

面试回答：

> 当前是本地 HTTP API + Python module 的实现，后续如果接入上层科研智能体，可以把 InstrumentAdapter 和 ExperimentStore 封装成 MCP tools/resources。

### Q：工具调用与 Skills 的关系？

关系是：

```text
Tool = 系统可调用的外部能力
Skill = 机器人任务语义层动作
```

例如：

- MuJoCo 是 tool/instrument。
- `push` 是 skill。
- `run_experiment(skill_plan)` 是工具调用。

## 6. 可能的面试方向

### 方向一：AI for Science 理解

#### Q1：AI for Science 和普通 AI 应用有什么区别？

推荐回答：

普通 AI 应用通常更关注用户体验、文本生成、推荐或业务流程效率；AI for Science 更强调可验证性、物理规律、实验成本、可复现性和可信结论。科学任务里，模型不能只给一个看起来合理的答案，还要能说明证据来源、实验条件、适用边界和下一步如何验证。

本项目对应到这个区别，就是没有只做一个“能说话的机器人助手”，而是把目标、协议、执行、指标、失败类型和重试记录都结构化。这样实验结果可以复查、可以比较，也可以成为下一轮实验的输入。

可能追问：

- 科学场景为什么不能只靠 LLM 生成结论？
- 仿真实验结果为什么仍需要可信评估？
- 如何避免小样本成功率被误读成泛化能力？

#### Q2：为什么科学发现需要 Agent？

推荐回答：

科学发现不是单次问答，而是长链路任务：理解目标、检索知识、设计实验、调用工具、执行计算或实验、分析结果、调整假设、进入下一轮。Agent 的价值在于把这些环节组织成连续工作流。

本项目里 Agent 的角色就是实验工作流编排器：它不直接控制每个关节，而是负责把自然语言实验目标转成 TaskSpec、ExperimentPlan 和 SkillPlan，调用 MuJoCo 执行，再基于 EvaluationReport 生成 RetryPlan，并触发一次 AutoRetry。

可能追问：

- 你的 Agent 和普通 pipeline 区别是什么？
- Agent 哪些环节是动态决策，哪些环节是固定工程逻辑？
- 为什么不用 LLM 直接控制实验设备？

#### Q3：实验闭环为什么重要？

推荐回答：

科学实验的价值不只是得到一次结果，而是通过结果不断修正假设和实验设计。如果系统只执行一次实验，不记录失败原因，也不生成下一轮建议，它就只是自动化脚本；只有能把执行结果回流到下一轮决策，才接近实验智能体。

本项目已经实现了最小闭环：

```text
TaskSpec -> MuJoCo Runs -> EvaluationReport -> RetryPlan -> AutoRetry -> ExperimentStore
```

这说明系统已经从“执行一次任务”前进到“根据失败自动做一次修正实验并记录经验”。

可能追问：

- AutoRetry 的触发条件是什么？
- 如果 Retry 后成功率没有提升，如何解释？
- 如何从一次 Retry 扩展为多轮自主实验？

#### Q4：数据回流在科学发现中的价值是什么？

推荐回答：

数据回流是 Agent 能持续变强的基础。没有数据回流，Agent 只能每次从零开始；有了结构化数据，系统才能知道哪些参数成功过、哪些失败类型最常见、下一轮应该扩大还是缩小变量范围。

本项目里数据回流包括：

- runs：每次实验的结构化结果；
- metrics：位移、接触步数、最大接触力、桌面安全；
- replay：轨迹回放；
- EvaluationReport：成功率、失败类型、样本数；
- ExperimentStore：历史实验记录。

可能追问：

- ExperimentStore 当前存什么？
- 历史记录如何用于下一轮规划？
- 如何设计长期实验记忆？

#### Q5：如何判断一个科学智能体输出是否可信？

推荐回答：

至少要看四点：

- 是否有明确输入协议和成功标准；
- 是否有可复现的执行过程；
- 是否有结构化观测和失败类型；
- 是否说明样本数、变量空间和适用边界。

本项目里，“推动 5cm”不会只看物体动没动，而是必须满足：

```text
abs(object_displacement - target_displacement_m) <= tolerance
and table_clearance_ok == true
```

这就是把可信评估放进系统，而不是靠主观观察。

### 方向二：Agent 工程架构

#### Q1：你的 Agent 架构怎么设计？

推荐回答：

我的 Agent 是一个受约束的实验工作流 Agent，而不是裸 LLM Agent。架构分成几层：

```text
User Goal
  -> Agent Orchestrator
  -> TaskSpec / ExperimentPlan / SkillPlan
  -> MuJoCo Instrument Adapter
  -> Runs / Metrics / Replay
  -> EvaluationReport
  -> RetryPlan / AutoRetry
  -> ExperimentStore
```

LLM 或规则解析器只负责高层目标理解和分析；真正执行前必须经过结构化协议、能力匹配、安全约束和物理仿真。

可能追问：

- Orchestrator 和普通后端 service 区别是什么？
- 哪些模块可以替换成外部 Agent 框架？
- 如果接真实机器人，架构哪里需要变？

#### Q2：Agent 和脚本的区别在哪里？

推荐回答：

脚本通常是固定输入、固定流程、固定输出。这个系统虽然工程上有 pipeline，但其中有多个动态决策节点：

- 从自然语言解析任务类型和成功标准；
- 根据任务生成实验变量；
- 选择机器人、工具和对象；
- 根据失败类型生成 RetryPlan；
- 自动执行一轮 revised task spec；
- 将结果写入实验记忆。

所以它不是单次脚本，而是一个带工具调用、结果反馈和重试决策的 agentic workflow。

可能追问：

- 如果 LLM 不可用，这还是 Agent 吗？
- fallback 规则会不会让它变成脚本？
- 当前智能性最强的环节在哪里？

#### Q3：如何管理上下文和状态？

推荐回答：

上下文分三层：

```text
任务上下文：goal、TaskSpec、ExperimentPlan、SkillPlan
执行上下文：runs、metrics、failure_type、replay
记忆上下文：ExperimentStore 历史实验记录
```

系统不会把所有信息都塞进一个 prompt，而是通过结构化对象流转。这样好处是上下文可压缩、可校验、可复现，也方便后续替换模型或执行器。

可能追问：

- 上下文是否共享？
- 状态如何流转？
- 三级记忆可以怎么设计？

#### Q4：如何防止 LLM 幻觉和错误工具调用？

推荐回答：

核心原则是不信任裸 LLM 输出。LLM 输出必须经过：

- TaskSpec 字段约束；
- robot/object/action 白名单；
- capability matching；
- success_criteria 后处理；
- MuJoCo 场景加载；
- 执行后物理指标验收。

例如 LLM 即使说任务成功，后端仍然按 `object_displacement`、`target_displacement_m`、`table_clearance_ok` 判定。成功与否不是模型主观判断。

可能追问：

- schema 校验做了哪些？
- 如果 LLM 输出不存在工具怎么办？
- 如何处理不可执行任务？

#### Q5：为什么没有直接用 LangChain / LlamaIndex？

推荐回答：

当前项目的核心难点不是搭通用聊天 Agent，而是定义实验协议、执行适配、安全验收和数据回流。LangChain 可以做工具编排，LlamaIndex 可以做知识检索，但 TaskSpec、SkillPlan、MuJoCo Adapter、EvaluationReport 这些领域结构仍然需要自定义。

所以当前先实现轻量 Orchestrator，保证核心闭环可控。后续如果要接入更多工具、文献检索、历史实验检索，可以把 LangChain/LlamaIndex 作为上层编排或 RAG 组件接入。

可能追问：

- LangChain 和 LlamaIndex 分别适合什么？
- MCP 能否用于这个项目？
- Skills 和 tools 的关系是什么？

### 方向三：Demo 项目深挖

#### Q1：这个 Demo 到底算不算 Agent？

推荐回答：

算，但要限定清楚：它是实验工作流 Agent，不是端到端机器人控制 Agent。它的智能性体现在高层实验流程：

```text
理解目标 -> 生成协议 -> 调用工具 -> 读取结果 -> 失败归因 -> 自动重试 -> 记录记忆
```

底层动作仍由动作原语、IK 和 MuJoCo 执行。这样设计是为了安全、可验证和可复现。

可能追问：

- 为什么不是机器人控制 Agent？

#### Q2：为什么用 MuJoCo？

推荐回答：

MuJoCo 在这个项目中不是为了展示动画，而是作为可编程虚拟实验仪器。它能提供接触、摩擦、刚体运动、物体位姿、末端状态和渲染帧。相比纯前端动画，它能产生可用于成功判定和失败归因的物理数据。

本项目把 MuJoCo 抽象成执行工具：

```text
ExperimentPlan / SkillPlan -> MuJoCo execution -> Runs / Metrics / Replay
```

#### Q3：AutoRetry 怎么触发？

推荐回答：

AutoRetry 的触发来自 EvaluationReport。如果当前样本中存在失败类型，系统会取 dominant failure，生成 RetryPlan，再根据失败类型修改 revised task spec 并执行一小轮修正实验。

例如 `table_penetration` 会触发：

- 提高接触/接近高度；
- 收紧桌面 clearance；
- 调整 push_force 搜索空间；
- 重跑一小轮实验；
- 对比原失败类型重试前后占比。

可能追问：

- Retry 后失败类型变化怎么解释？

- 如何扩展成多轮 retry？

#### Q4：ExperimentStore 存什么？

推荐回答：

ExperimentStore 当前写入 `results/nlp_experiment_store.jsonl`，每轮 NLP 实验保存：

- task_spec；
- robot_id；
- experiment_plan；
- skill_plan；
- evaluation_report；
- retry_plan；
- retry_execution；
- recorded_at。

它的作用是让实验不再是一次性输出，而是可以被后续检索、统计和复用。

#### Q5：成功率为什么可信？

推荐回答：

成功率可信不是因为数字高，而是因为成功标准明确、样本数明确、失败原因可追溯。以推动 5cm 为例，成功必须同时满足目标位移误差和桌面安全：

```text
abs(object_displacement - target_displacement_m) <= tolerance
and table_clearance_ok == true
```

因此移动 30cm 会被判为 `overshoot`，移动不足会被判为 `undershoot`，违反桌面安全会被判为 `table_penetration`。

可能追问：

- 100% 成功率说明什么？
- 样本少怎么办？
- strict success 和 partial success 如何区分？

#### Q6：如果现场 Demo 失败怎么办？

推荐回答：

失败不是展示事故，而是实验智能体的展示点。系统会输出 failure_type、位移误差、接触步数、table_clearance_ok 和 RetryPlan。可以现场解释失败属于哪一类，并展示 AutoRetry 如何修正实验。

这个项目的目标不是隐藏失败，而是把失败结构化，让系统知道下一步怎么试。

### 方向四：机器人/仿真/控制

#### Q1：当前动作原语有哪些？

推荐回答：

动态管线支持 8 个基础动作原语：

```text
reach / grasp / lift / place / push / insert / release / wait
```

FR3 技能层还有：

```text
reach_touch / button_press / pick_lift / contact_sweep / tool_contact_sweep / peg_insert
```

这些动作原语是当前 Demo 的可执行边界。Agent 不能随意生成任意动作，必须落到已注册技能或动作原语中。

#### Q2：IK 怎么实现？

推荐回答：

底层使用 Jacobian 阻尼最小二乘 IK：

```text
dq = J^T (J J^T + lambda^2 I)^-1 e
```

其中 `J` 是末端 Jacobian，`e` 是末端误差，`lambda` 是阻尼项，用于提升奇异点附近稳定性。MuJoCo 提供关节状态和 Jacobian，系统把任务空间目标转成关节增量执行。

可能追问：

- 为什么要阻尼？
- IK 失败怎么办？
- 这个方法适合哪些任务？

#### Q3：为什么之前会出现机械臂陷进桌子？

推荐回答：

根因是成功判定和物理安全最开始没有统一。如果只看物体是否移动，就可能出现物体被推动了，但机械臂几何体和桌面发生不合理接触，仍然被算成功。

现在已经补强：

- 物体初始高度按几何尺寸放到桌面上方；
- 成功标准加入目标位移；
- 执行日志记录 `table_clearance_ok`；
- 记录 `min_ee_z`、`table_contact_steps`、`table_contact_pairs`；
- 违反桌面安全会归因为 `table_penetration`，不能算成功。

可能追问：

- 这是不是完整碰撞规划？
- 为什么 AutoRetry 后从 table_penetration 变成 undershoot？
- 如何进一步避免全身碰撞？

#### Q4：如何做全局避障？

推荐回答：

当前系统还不是完整全局规划器。下一步会把 MotionPlan 层升级为可插拔运动规划接口，接入：

- MoveIt / OMPL 做采样式路径规划；
- TrajOpt 做约束轨迹优化；
- signed distance field 做碰撞代价；
- 连续碰撞检测避免运动过程中穿透。

上层 TaskSpec、SkillPlan 和 EvaluationReport 不需要推倒重来，只替换底层 MotionPlanner。

#### Q5：仿真和真实差距怎么处理？

推荐回答：

仿真结果不能直接等同真实成功。真实机械臂会有标定误差、摩擦差异、控制延迟、传感噪声和安全风险。

迁移路径是：

```text
TaskSpec / SkillPlan
  -> SimAdapter(MuJoCo)
  -> RealRobotAdapter(真实机械臂 SDK)
  -> ExecutionTrace
  -> EvaluationReport
```

真实执行前必须加入工作空间检查、速度/力限制、碰撞预检查、急停和人工确认。

### 方向五：工程能力

#### Q1：前后端怎么通信？

推荐回答：

前端是 `web_demo/` 实验控制台，后端是 `mujoco_bridge.server` 提供本地 HTTP API。主要入口包括：

- `/health`
- `/api/robots`
- `/api/tasks`
- `/api/nlp/run`
- `/api/nlp/status`
- `/api/run_experiments`

NLP 管线运行时，前端提交目标和参数，后端执行解析、仿真、评估、重试和记忆写入，再把完整 JSON 返回给前端展示。

#### Q2：如何验证系统可靠性？

推荐回答：

项目有三层验证：

```powershell
.\.venv\Scripts\python.exe -m mujoco_bridge.verify_strict_constraints
.\.venv\Scripts\python.exe -m mujoco_bridge.verify_nlp_pipeline
.\.venv\Scripts\python.exe -m mujoco_bridge.verify_roadmap
```

分别验证：

- 严格目标位移和桌面安全；

- 协议、技能、能力、评估和存储链路。

#### Q3：如何部署给别人看？

推荐回答：

有两种方式：
